## Executive Summary

Mahina v0.1 ended at a deterministic but visually inert state: `luna-init` supervises services, `luna-splash` paints a framebuffer boot screen, and the system drops to a BusyBox shell. v0.2 is the milestone in which Mahina grew a display server, a native GUI toolkit, and the first generation of graphical applications on top of it. Across six commits since the prior snapshot, the repository added a DRM/KMS compositor with surface compositing and now window-management messages, a ~1,500-line LunaGUI widget toolkit, an ANSI/VT100 terminal emulator, a ten-page installer UI, six core desktop applications (settings, files, calculator, text viewer, system info, task manager), and — in the commit that closes this milestone — a window manager (`luna-shell`) that registers with the compositor as a privileged client, places and focuses application surfaces, and binds two global hotkeys.

That is the intended shape of v0.2. What actually builds today is narrower. Two of LunaGUI's core source files (`core/application.c`, `core/window.c`) fail to compile against the project's own headers because a required private header is never included; three newly-added source files in two different subsystems (the compositor's `input/keyboard.c` and `protocol/wm.c`, and LunaGUI's `widgets/canvas_widget.c` and `widgets/scroll.c`) are never added to their respective `Makefile.inc` source lists, so even once the compile errors are fixed, `lgp-compositor`, `luna-shell`, and `luna-terminal` fail to *link*. A text-rendering function is declared with one signature in the shared private header and defined with a different, incompatible signature in its implementation file — confirmed by this audit to produce a guaranteed segmentation fault at the first call, once the build and a font asset are both present. No `.psf` font file is shipped anywhere in the repository, so even past those two defects, every label and button caption renders as nothing. The disk-image builder copies only `luna-shell` and `luna-terminal` of the ten new application binaries into the bootable image. None of this new surface area has unit-test or CI coverage.

None of this is a moral judgment on the work — a large amount of real systems code landed in one day, including a from-scratch ANSI parser, a working surface-compositing pipeline with alpha blending, and a complete window-manager protocol extension. It is a statement about what is verified to run today versus what is written. The remainder of this document separates the two carefully.

## Goals of v0.2

The following objectives were carried out, in the sense that the corresponding code exists and (with the exceptions detailed in §Technical Debt) compiles or has been demonstrated to compile in isolation:

- **LunaGUI maturation** — pointer input routing (motion + button → hit-test → click callback), a real "Mahina Design System" renderer replacing placeholder drawing, recursive widget-tree teardown (`lgui_widget_destroy()`), explicit widget sizing, and an expanded public API.
- **LGP protocol expansion** — a window-manager message family (`LGP_MSG_WM_SURFACE_CREATED/DESTROYED/SET_SURFACE_POSITION/SET_FOCUS/SET_STATE/GRAB_KEY`), a clipboard message pair (`LGP_MSG_CLIPBOARD_SET/GET/DATA`), and keyboard-event message types, added to `tlv.h`'s message catalog.
- **Window compositor improvements** — the surface compositor now accepts WM-issued position changes and re-composites on demand; the existing Z-order/ARGB8888 alpha-blending path from the previous sprint is unchanged and still in use.
- **Surface management** — unchanged from the previous milestone (`CREATE_SURFACE`/`DESTROY_SURFACE`/`COMMIT_BUFFER`), now consumed by a second, independent client class (the window manager) in addition to ordinary application clients.
- **Keyboard routing** — new this milestone: an evdev-based keyboard reader in the compositor (`input/keyboard.c`), per-surface keyboard focus tracking, a WM key-grab mechanism for global hotkeys, and client-side decoding in LunaGUI's event loop.
- **Widget focus** — click-to-focus via hit-testing, with `app->focused_widget` receiving routed key events, and a separate WM-driven `lgui_wm_set_focus()` channel that changes which *surface* (not widget) receives compositor-level keyboard focus.
- **Window manager foundation** — new this milestone: `luna-shell` registers as a window manager (`LGP_CAP_WINDOW_MANAGER`, root-only), cascades new application windows, tracks them in a flat array, and implements Super+T (launch terminal) and Alt+Tab (cycle focus) as global hotkeys.
- **luna-shell** — implemented as described above, with a wallpaper surface and a top-bar surface (logo label + a static, non-live clock label).

Two items on the candidate list were **not** meaningfully advanced this milestone and should not be read as done: **terminal rendering** remains text-only (the comment in `luna-terminal/main.c` documents pixel-cell rendering as still deferred), and **scroll containers** exist as a widget type with clipped rendering but have no input path that ever changes their scroll offset — see §LunaGUI for detail. **Clipboard** is implemented but is a single unbounded global slot with no capability enforcement, also detailed below.

## Major Engineering Milestones

### LGP Compositor — Window Manager & Keyboard Extension

**Purpose.** Extend the existing DRM/KMS compositor (`lgp-compositor`, in place since the prior milestone) with a privileged window-manager channel and a second input device class (keyboard), so that a userland process can position, focus, and state-manage other clients' surfaces instead of every surface being placed at a fixed origin.

**Architecture.** The WM channel is a capability, not a separate protocol: a client that requests and is granted `LGP_CAP_WINDOW_MANAGER` (bit 7, root-only per `lgp_caps_negotiate()`) gains access to six new message types defined in `protocol/wm.h`/`wm.c`. The compositor tracks at most one `state->wm_client` and forwards `WM_SURFACE_CREATED`/`WM_SURFACE_DESTROYED` notifications to it whenever any client creates or destroys a surface. The WM can then send `SET_SURFACE_POSITION` (updates `surface->x/y` and triggers a recomposite), `SET_FOCUS` (changes `state->keyboard_focus_session_id`), `SET_STATE` (accepted and stored but not yet acted on — no minimize/maximize/fullscreen behavior exists), and `GRAB_KEY` (registers a global hotkey in a fixed 16-entry table that takes priority over per-surface keyboard focus). Keyboard input itself comes from a new `input/keyboard.c` module that scans `/dev/input/event0`–`event9` for the first device reporting `KEY_A` support, tracks Shift/Ctrl/Alt/Super modifier state, and forwards `LGP_MSG_KEYBOARD_KEY` either to the grabbing WM client or to whichever surface currently holds keyboard focus.

**Implementation Summary.** `src/lgp-compositor/protocol/wm.c` (69 lines) implements little-endian TLV encode/decode for the six WM messages. `src/lgp-compositor/input/keyboard.c` (85 lines) implements evdev scanning, modifier tracking, and dispatch. `main.c` gained ~225 lines wiring both into the epoll event loop and the message-handling switch, plus the four `lgp_handle_wm_*` handler functions. Click-to-focus was added to the existing pointer-button dispatcher: pressing a button over a surface now also sets keyboard focus to that surface's owning session, independent of the WM's explicit `SET_FOCUS` channel.

**Important files.** `src/lgp-compositor/protocol/wm.{c,h}`, `src/lgp-compositor/input/keyboard.{c,h}`, `src/lgp-compositor/main.c` (message dispatch, click-to-focus, keyboard fd registration), `src/lgp-compositor/protocol/caps.{c,h}` (capability bit), `etc/luna/services/luna-shell.toml` (new service definition).

**Current maturity.** Functionally complete as written, but **does not link**. `keyboard.c` and `wm.c` are never added to `LGP_COMPOSITOR_SOURCES` in `src/lgp-compositor/Makefile.inc` (the list was last touched when the compositor was first scaffolded and was not updated in this milestone's commit). `main.c` calls nine functions — `lgp_keyboard_init`, `lgp_keyboard_get_fd`, `lgp_keyboard_pump`, `lgp_wm_encode_surface_created`, `lgp_wm_encode_surface_destroyed`, `lgp_wm_decode_set_position`, `lgp_wm_decode_set_focus`, `lgp_wm_decode_set_state`, `lgp_wm_decode_grab_key` — that are defined only in these two unbuilt files. Both files were confirmed by this audit to compile cleanly in isolation (`gcc -std=c17 -c`, zero errors); the defect is purely a missing build-system entry, not a defect in the files themselves. See §Technical Debt (Critical) and §Verification.

**Why it matters.** Window placement and keyboard input are the two capabilities every subsequent desktop feature depends on — text entry, window dragging, multi-window focus, and any future LunaGUI scroll/keyboard work all sit downstream of this. Until the build-system gap is closed, none of it is reachable on a running system; the compositor binary itself cannot be produced from a clean checkout via `make lgp-compositor` or `make all`.

### LunaGUI Toolkit — Input Routing, Renderer Rewrite, WM Client API

**Purpose.** LunaGUI is Mahina's only GUI toolkit — no GTK, Qt, or other third-party framework is used anywhere in the tree. It is the library every application in this release, including `luna-shell` itself, links against.

**Architecture.** A `lgui_application_t` owns one connection to `lgp-compositor`, performs the `LGP_HELLO` handshake, and runs a `poll()`-based event loop (the compositor socket plus up to eight user-registered file descriptors, 16 ms timeout — effectively a 60 Hz tick). Incoming TLV frames are decoded inline in the event loop and dispatched: pointer motion updates `cursor_x/cursor_y`; pointer button-down performs a recursive hit-test (`lgui_widget_hittest()`, front-to-back, deepest match wins) against every window's widget tree and invokes `on_click`; keyboard-key events go to both a registered global-key callback (for window-manager-style global hotkeys) and the currently focused widget's `on_key`. A widget is a flat tagged struct (`int type` selecting Label/Button/VBox/HBox/Scroll/Canvas, fixed 32-child array, 256-byte text buffer) rather than a class hierarchy, rendered by a single recursive `lgui_widget_render()` switch in `render.c` against a hand-rolled six-color "Mahina Design System" palette.

**Implementation Summary.** This milestone rewrote `render.c` (50 → 187 lines) with the design-system palette and VBox/HBox flex layout; added pointer hit-testing and dispatch to `application.c`; added `lgui_widget_destroy()` for recursive tree teardown (closing a real leak across the page-navigation pattern every multi-page app in this release uses); and, in the window-manager commit, added a window-manager client API (`lgui_application_create_wm()`, `lgui_wm_set_surface_position/focus/state/grab_key()`), a keyboard-event decode path, and a `core/keyboard.c` module implementing `lgui_keymap_translate()` (scancode → ASCII, no Unicode/compose support, documented in its own header as a known gap).

**Important files.** `src/luna-gui/core/{application,window,keyboard}.c`, `src/luna-gui/render/{render,canvas,font}.c`, `src/luna-gui/widgets/{widget,button,label,vbox,hbox,scroll,canvas_widget}.c`, `src/luna-gui/include/{lunagui,lgui_private,widget_private,window_private}.h`.

**Current maturity — does not currently compile or link as committed.** This audit identified three independent defects, each confirmed by direct compilation against the project's own header layout:

1. **`core/application.c` and `core/window.c` do not compile.** Neither file includes `widget_private.h`, the only header that defines the full `struct lgui_widget_t`. Both files dereference widget fields directly (`hit->type`, `hit->on_click`, `w->child_count`, `root->width`, etc.), which requires the complete type. Compiling each file in isolation against the real include path (`gcc -std=c17 -Isrc/luna-gui/include -Isrc -c`) produces 13 and 4 "invalid use of incomplete typedef" errors respectively — 17 in total. This was also reproduced by invoking the project's actual `make lunagui` rule directly (with `CC=gcc` substituted for the unavailable `clang`), which fails at exactly this point.
2. **`widgets/canvas_widget.c` and `widgets/scroll.c` are never added to `LUNAGUI_SOURCES`** in `src/luna-gui/Makefile.inc`. Both compile cleanly in isolation. But `luna-shell` and `luna-terminal` both call `lgui_canvas_widget_create()`/`lgui_canvas_widget_set_render()` (for wallpaper and terminal-grid rendering respectively), and would fail to link against `liblunagui.a` as currently built. `luna-terminal` additionally calls `lgui_keymap_translate()`, defined in `core/keyboard.c` — also absent from `LUNAGUI_SOURCES`.
3. **`lgui_font_draw_text()` has two incompatible signatures across translation units.** `include/lgui_private.h` declares it as `(lgui_canvas_t *canvas, int x, int y, const char *text, uint32_t color)` — five arguments — and `render/canvas.c` calls it through that declaration. `render/font.c` *defines* the same symbol as `(void *pixels, int x, int y, uint32_t stride, const char *text, uint32_t color)` — six arguments, with the fourth and fifth reinterpreted. Because `font.c` never includes `lgui_private.h`, neither file's compiler ever sees the conflict, so both compile and the mismatch links silently (C does not name-mangle by signature). This audit built a minimal reproduction linking the real `canvas.o` and `font.o` against a synthetic caller with a valid embedded PSF-format font supplied: the call **segfaults** (process exit code 139 / SIGSEGV) the first time a caller reaches the `while (*text)` loop in `font.c`, because the argument the function treats as a `const char *` was actually passed as a `uint32_t` color value by the caller. With no font present (the actual state of the repository — see next item) the bug is currently masked by an earlier `if (!g_font_data) return;` guard, which is why it has not yet been observed.

**Independently, no `.psf` bitmap font ships anywhere in this repository**, and `scripts/build-image.sh` never installs one to `/usr/share/fonts/`. `lgui_font_init()` silently no-ops when the file is missing. The net effect, even setting the signature bug aside: every `lgui_canvas_draw_text()` call in this codebase — every label, every button caption, in every one of the ten new applications — currently renders nothing. Fixing the missing-asset problem without first fixing the signature mismatch would change a silent no-op into a crash.

**Why it matters.** This is the toolkit every other v0.2 deliverable is built on. The defects above are concentrated in exactly the two files (`application.c`, `window.c`) and one function (`lgui_font_draw_text`) that every single application and `luna-shell` itself depends on, so they are not isolated to one feature — they block the entire milestone's GUI surface from running until corrected.

### Window Manager & Desktop Shell — `luna-shell`

**Purpose.** Provide the first process that owns desktop-level policy: where new windows appear, which surface has keyboard focus, and what global keyboard shortcuts exist — responsibilities the compositor deliberately does not take on itself (it is described in its own source comments as mechanism, not policy).

**Architecture.** `luna-shell` is a 152-line single-file LunaGUI client that calls `lgui_application_create_wm()` instead of the ordinary `lgui_application_create()`, which performs the LGP handshake requesting `LGP_CAP_WINDOW_MANAGER` in addition to the capabilities every client requests. It registers two callbacks (`on_surface_created`, `on_surface_destroyed`) that the compositor invokes whenever any client's surface lifecycle changes, and a global-key callback for hotkeys. On startup it creates two of its own surfaces — a wallpaper at layer `LGUI_LAYER_WALLPAPER` (a flat `0xFF2A2A35` fill via a canvas widget) and a top bar at layer `LGUI_LAYER_SHELL` (an HBox with a "MahinaOS" label and a clock label) — then calls `lgui_wm_grab_key()` twice, for Super+T and Alt+Tab.

**Implementation Summary.** When any client creates an `LGP_SURFACE_APPLICATION_WINDOW`-type surface, `luna-shell` records it and cascades its position (30px steps from a base of 100,100, wrapping after x>400) via `lgui_wm_set_surface_position()`, then calls `lgui_wm_set_focus()` to give it keyboard focus immediately. Super+T forks and `execvp`s `luna-terminal`. Alt+Tab rotates the internal surface array and refocuses whichever surface is now at the tail — a simple, working round-robin, though with no visual indication of which window is focused (no border highlight or similar) and no handling for the case where the focused window itself is the one rotated out.

**Important files.** `src/luna-shell/main.c`, `etc/luna/services/luna-shell.toml` (new service definition: `after = ["lgp-compositor"]`, `restart.policy = "always"`, runs as root).

**Current maturity.** Logic is straightforward and, by inspection, correct for what it attempts. It cannot currently run: it depends on `liblunagui.a`, which does not build (see previous section), and even if it did, it calls `lgui_canvas_widget_create()`/`lgui_canvas_widget_set_render()` for its wallpaper, which are in the set of LunaGUI symbols missing from the static library's source list. The top-bar clock label is initialized once to the literal string `" 12:00 "` and is never updated — there is no timer wiring it to wall-clock time, unlike the separate (and not currently wired up) `luna-desktop` dock, which does have a working 60-second `timerfd`-driven clock. See the next paragraph and §Technical Debt for the resulting duplication.

**A second, overlapping shell binary exists and is not reconciled with this one.** `src/luna-desktop/main.c`, scaffolded in the commit immediately prior to this milestone's WM work, independently implements a wallpaper surface and a dock/panel surface with three placeholder buttons (Menu/Files/Terminal — none wired to `on_click`) and a working live clock. It is **not** a window-manager client (plain `lgui_application_create()`, no WM capability request), is **not** declared in `etc/luna/services/` (no `luna-desktop.toml` exists), and is **not** copied into the disk image by `build-image.sh`. As committed, `luna-desktop` is dead weight relative to `luna-shell` — it builds (modulo the toolkit-wide breakage above) but has no path to ever run on a booted system, while `luna-shell` is the one wired into the service graph and the image. Neither this document nor the repository's own status files designate one as superseding the other; this should be resolved explicitly in v0.3 rather than left as two parallel "shell" implementations.

**Why it matters.** This is the headline feature of the milestone — the first process in Mahina's history that exercises compositor-level policy rather than just drawing into a single surface. Its correctness (cascading placement respects the existing Z-order/alpha-blend compositing path; click-to-focus and WM `SET_FOCUS` do not conflict) is good evidence that the window-manager protocol design in `wm.c`/`wm.h` is sound. The fact that it cannot currently be exercised end-to-end is a build-system and toolkit problem, not a design problem with the WM channel itself.

### luna-terminal — ANSI/VT100 Terminal Emulator

**Purpose.** A real terminal application — the first one — replacing the BusyBox `sh` fallback as the intended primary command-line surface inside the GUI, reachable via `luna-shell`'s Super+T hotkey.

**Architecture.** `ansi.c`/`ansi.h` implement a small explicit state machine (`GROUND → ESCAPE → CSI → CSI_PARAM`) over a `term_cell_t` grid (per-cell foreground/background color in XRGB8888, bold flag, dirty flag), with a 2000-line circular scrollback buffer. `pty.c` spawns `/bin/sh` via `forkpty()`; the PTY's master fd is registered with `lgui_application_add_fd()` so output is fed through `ansi_feed()` into the grid as it arrives, and `TIOCSWINSZ` keeps the shell's reported column width in sync with the displayed grid via `term_grid_resize()`.

**Implementation Summary.** Supported escape sequences: cursor movement (`A`/`B`/`C`/`D` relative, `H`/`f` absolute 1-indexed), erase-in-display and erase-in-line (`J`/`K` modes 0/1/2), SGR color (8 base + 8 bright via 30–37/40–47/90–97/100–107, plus bold and reset), and cursor visibility (`?25h`/`l`). Window is sized to 80×24 characters (648×392 px at 4px padding). Clean teardown sends `SIGHUP` to the shell on application exit.

**Important files.** `src/luna-terminal/{main,pty,ansi}.c`, `src/luna-terminal/{pty,ansi}.h`.

**Current maturity.** The ANSI parser and PTY plumbing are real and, by code inspection, handle the documented escape subset correctly. Two gaps remain by the code's own design: rendering is **text-only** — the grid is never blitted through `lgui_window_get_canvas()` into actual pixels with per-cell color, despite that function existing in the public API specifically for this purpose — and there is **no keyboard input path into the shell** at all; typed characters never reach the PTY, because (as with every other application in this release) the keyboard-event plumbing that landed in this same milestone's compositor work was never connected on the LunaGUI client side for text widgets. The terminal also depends on the toolkit-wide build defects described above (`liblunagui.a` won't build; `lgui_canvas_widget_create`/`lgui_keymap_translate` are called but unlinked) and is one of only two application binaries (`luna-shell` is the other) that `build-image.sh` actually copies into the disk image — meaning it is, ironically, the most "deployed" of the new applications while also depending on the most build-broken parts of the toolkit.

**Why it matters.** Once the build is fixed and keyboard routing is connected, this becomes Mahina's primary interactive shell inside the GUI, displacing the BusyBox fallback shells as the documented intended UX. Today it cannot be typed into even if it could be launched.

### luna-installer — Graphical Installer (UI only)

**Purpose.** A ten-page graphical installation flow, expanded this milestone from an earlier three-page stub.

**Implementation Summary.** Welcome → Language → Keyboard → Timezone → Disk Selection → Partitioning Summary → User Creation → Summary → Installation Progress → Complete/Reboot. Each page transition calls `lgui_widget_destroy()` on the outgoing tree before building the next, consistent with the leak-safe pattern used across every multi-page application in this release.

**Current maturity.** UI flow is complete and internally consistent by inspection. Installation progress is **simulated** — there is no `mkfs`, partitioning, or filesystem-copy code behind the "Installing" page; clicking through to "Complete" does not modify any disk. The language-selection page also has a known, documented gap: the selected language is never stored anywhere (a literal `TODO` in source), so the choice is presentationally collected but has no effect.

**Important files.** `src/luna-installer/main.c` (442 lines).

**Why it matters.** A complete, navigable installer UI is a prerequisite for any future "boot the ISO and install to disk" workflow, but as it stands the application cannot actually install Mahina onto a machine — running it through to completion and rebooting would simply boot back into the same live/dev environment.

## Boot Pipeline

```
UEFI
  │
  ▼
Limine                 boot/limine.conf — loads vmlinuz-mahina + initramfs-mahina.img
  │
  ▼
Linux Kernel            6.9.7 per scripts/build-kernel.sh (kernel/.config.notes still
  │                      documents "6.6.x LTS" — unresolved mismatch, unchanged since
  │                      the prior milestone, not touched in v0.2)
  ▼
Initramfs                statically-linked luna-init, busybox, minimal /dev nodes
  │
  ▼
luna-init (PID 1)
  │
  ├─ Stage 0–3: early mounts, signalfd/epoll init, fstab.toml mounts, hostname
  ├─ Stage 4: load etc/luna/services/*.toml → Kahn's-algorithm dependency graph
  │            → async supervisor starts dbus/networkmanager/ntpd/pipewire/
  │            wireplumber/luna-shell* (lgp-compositor explicitly excluded from
  │            this pass — see below). *luna-shell is PENDING here but blocked:
  │            its only dependency (lgp-compositor) is held at STOPPED until Stage 5.
  ├─ [boot-complete check, 3.5s splash hold]
  ├─ Stage 5: splash_stop() → supervisor_start_one("lgp-compositor") (manual,
  │            bypassing the Stage 4 graph) → luna_log_switch_to_runtime()
  │            → welcome banner → forks BusyBox/sh on stdout AND on /dev/tty1
  └─ Stage 6: UNGATED. luna-init does not wait for lgp-compositor to become
               ready, and does not wait for luna-shell to start, before dropping
               to the two interactive shells above. Once lgp-compositor's
               readiness socket (/run/lgp/compositor.sock) appears, the async
               supervisor's periodic pump (driven by main.c's 100ms timerfd)
               independently notices luna-shell's dependency is satisfied and
               starts it in the background — but this happens concurrently
               with, not before, the shell drop.
```

This is unchanged in its ungated nature from the prior milestone's documented finding (`MAHINA_ENGINEERING_SNAPSHOT.md` §1.3 flagged the same gap against an earlier compositor-only Stage 5). Adding `luna-shell` as a dependency-graph service this milestone did not add a corresponding wait — Stage 6 still proceeds unconditionally. This is consistent with the project's own documented design intent ("Mahina setup as GUI-first with terminal setup reserved for emergency and development use," per `CHANGELOG.md`'s Unreleased section) in the sense that the two text shells are an intentional fallback rather than the primary UX, but it does mean a booting system currently has no signal — visual or otherwise — for whether the desktop session actually came up.

## Graphics Stack

**DRM/KMS.** Unchanged this milestone: legacy (non-atomic) `drmModeSetCrtc` modesetting via a dumb buffer, opened against `/dev/dri/card0`. `drm_device_t.atomic_supported` is detected but never branched on.

**LGP (Luna Graphics Protocol).** A from-scratch, non-Wayland Unix-socket protocol at `/run/lgp/compositor.sock` (mode `0660 root:video`), framed with a 6-byte TLV header (`uint16_t type`, `uint32_t length`, per DL-053). The message catalog as of this milestone, from `protocol/tlv.h`:

| Range | Messages | Status |
|---|---|---|
| `0x0001–0x0002` | `HELLO` / `HELLO_REPLY` | Implemented, unchanged |
| `0x0010` | `FILL_RECT` | Implemented, capability-gated (`LGP_CAP_DIRECT_LGP`) since the prior milestone |
| `0x0100–0x0103` | `CREATE_SURFACE` / reply / `DESTROY_SURFACE` / `COMMIT_BUFFER` | Implemented, unchanged |
| `0x0110–0x0111` | `POINTER_MOTION` / `POINTER_BUTTON` | Implemented, unchanged. Encoded big-endian. |
| `0x0112` | `KEYBOARD_KEY` | **New this milestone.** Implemented (compositor-side dispatch and client-side decode both present). Encoded big-endian. |
| `0x0113` | `POINTER_SCROLL` | **Defined, never implemented.** No compositor code encodes it (the raw PS/2 mouse reader in `input/mouse.c` only ever parses 3-byte packets and has no scroll-wheel byte handling), and no client decodes it. Dead protocol constant. |
| `0x0120–0x0122` | `CLIPBOARD_SET` / `GET` / `DATA` | **New this milestone.** Implemented as a single global string slot (`state->global_clipboard`) with no per-client isolation and no capability check — see §Technical Debt. Encoded big-endian for `DATA`. |
| `0x0200–0x0205` | `WM_SURFACE_CREATED/DESTROYED/SET_SURFACE_POSITION/SET_FOCUS/SET_STATE/GRAB_KEY` | **New this milestone.** Implemented; encoded **little-endian**, inconsistent with the big-endian encoding used by every other multi-byte LGP message family. Both compositor and client agree with each other per-type, so this is not a functional bug, but it is an unexplained inconsistency in a protocol whose wire format was deliberately frozen by Decision Log entry DL-053 last milestone. |
| `0xFFFF` | `ERROR` | Unchanged |

**Surface lifecycle.** Unchanged: `CREATE_SURFACE` validates requested width/height and layer against the live display mode and the requesting client's granted capabilities (`lgp_surface_validate_request()`); `COMMIT_BUFFER` accepts an `SCM_RIGHTS`-passed shared-memory `XRGB8888`/`ARGB8888` buffer. New this milestone: the compositor now also notifies a registered WM client of every surface's creation and destruction via `WM_SURFACE_CREATED`/`DESTROYED`.

**Buffer commits / shared memory.** Unchanged. `memfd`-backed buffers passed via `SCM_RIGHTS`; a bounded per-connection receive buffer reassembles TLV frames split across socket reads.

**Page flips.** Unchanged: legacy `drmModePageFlip`, full front-buffer recomposite on every commit (no damage tracking — see §Performance).

**Socket lifecycle.** Unchanged: one Unix-domain listener, per-connection client objects, `SO_PEERCRED`-gated capability negotiation at `HELLO` time.

**Capability model.** `LGP_CAP_WINDOW_MANAGER` (bit 7) was added this milestone, granted only to `uid == 0` clients, matching `luna-shell.toml`'s `identity.user = "root"`. Two gaps identified by this audit:

1. `WM_SET_SURFACE_POSITION`, unlike `CREATE_SURFACE`, performs **no bounds validation** against the live display dimensions — `lgp_handle_wm_set_position()` writes the WM-supplied `x`/`y` directly into the surface struct. `lgp_surface_manager_composite()` then blits that surface with no per-surface clipping against the destination buffer. A WM client that supplied an out-of-range position (whether through a bug or a compromised process) would cause `lgp_surface_manager_composite()` to read and write outside the mapped DRM front-buffer region. `luna-shell`'s own cascading-placement logic stays within safe bounds, so this is latent rather than triggered, but it is a new privileged input path introduced this milestone without the bounds-checking discipline `CREATE_SURFACE` already has.
2. `lgp_caps_negotiate()` explicitly denies `LGP_CAP_CLIPBOARD` for every client (comment: *"clipboard [is] not implemented yet, so deny"*) — but the `CLIPBOARD_SET`/`CLIPBOARD_GET` message handlers added this same milestone perform **no capability check at all**; any client, granted or not, can read or overwrite the global clipboard. The capability bit and the actual enforcement diverged within the same commit sequence that introduced both.
3. The four new WM-only message handlers (`lgp_handle_wm_set_position/focus/state`, `lgp_handle_wm_grab_key`) each test `client->caps_granted & 128` directly rather than the named `LGP_CAP_WINDOW_MANAGER` constant already available via the `caps.h` include in the same file — functionally correct (128 == `1u << 7`) but a maintainability regression against the rest of the file's style.

## LunaGUI

**Widget system.** A flat, tagged `struct lgui_widget_t` (type 1–6: Label, Button, VBox, HBox, Scroll, Canvas), fixed 32-child array, no dynamic widget tree beyond that ceiling.

**Canvas.** `lgui_canvas_t` wraps a raw `XRGB8888` pixel buffer with a 16-deep clip-rect stack (`push_clip`/`pop_clip`, intersected against the current top on push). `fill_rect` and `draw_rect_outline` are implemented and clip-respecting. `draw_text` is implemented but, per the analysis above, both functionally inert (no font asset shipped) and latently crash-inducing (signature mismatch) once a font is supplied.

**Focus.** Two independent focus concepts exist and do not always agree: widget-level focus (`app->focused_widget`, set by click-to-hit-test, receives `on_key` callbacks) and compositor-level surface focus (`state->keyboard_focus_session_id`, settable either by clicking a surface or by the WM's explicit `SET_FOCUS`). For a single-window application these coincide in practice; for `luna-shell` (which manages many surfaces' compositor-level focus but has its own separate widget tree) they are genuinely distinct, by design.

**Rendering.** Full-tree, top-down, every dirty window, every event-loop tick (see §Performance). VBox/HBox compute layout immediately before rendering their children (no separate layout pass).

**Layout.** Fixed-padding VBox (4px top, 8px left margin per child) and HBox (left-to-right, default per-widget sizing). No flex-grow, no percentage sizing, no min/max constraints.

**Events.** Pointer motion and button-down are fully routed (motion updates cursor position; button-down hit-tests and fires `on_click`). Button-up is read by the compositor's mouse driver but the client-side decode only acts "on button-down only" per its own comment — button-up is decoded but produces no widget-level effect. Keyboard events are new this milestone and are routed to both a global-key callback and the focused widget's `on_key`, but **no widget type in this release actually implements `on_key`** — `lgui_widget_set_on_key()` exists in the public API and is wired into the dispatch path, but no Label, Button, or other shipped widget registers a callback through it, and no text-input widget exists at all. The keyboard-routing infrastructure is therefore complete on the compositor and toolkit-dispatch side but has no consumer yet.

**Scroll.** `lgui_scroll_container_create()`/`set_child()` exist, and `render.c` correctly clips and offsets a scroll container's single child by `scroll_offset_x/y` during rendering. However, `scroll_offset_x/y` is **never written anywhere in the codebase** outside its own struct definition — there is no scroll-wheel input path (consistent with `POINTER_SCROLL` being a dead protocol constant, above) and no programmatic API to set it. The widget type exists and renders correctly but is permanently static. `luna-files` and `luna-text` both contain comments anticipating its use ("once `lgui_scroll_container` is available") despite the type already existing in source — those comments were written against an earlier state of the tree and were not updated when `scroll.c` landed, nor would updating them currently help, since `scroll.c` is also absent from `LUNAGUI_SOURCES` (see §Major Engineering Milestones).

**Clipboard.** `lgui_clipboard_set_text()`/`request_text()` exist client-side and round-trip correctly against the compositor's global single-slot store, with the capability-enforcement gap noted in §Graphics Stack.

**Input routing.** Pointer: complete and demonstrated correct by code inspection (hit-test recursion order is children-first, front-to-back, matching expected Z-order semantics). Keyboard: compositor-to-client wire path complete; client-to-widget dispatch path complete; no consuming widget exists yet.

**Current limitations (this milestone, in order of severity).** Does not compile (`application.c`, `window.c` missing-include defect). Does not link even once compiled (`canvas_widget.c`, `scroll.c`, `core/keyboard.c` absent from the build). Text rendering will segfault once the prior two are fixed and a font asset is supplied, due to the `lgui_font_draw_text` signature mismatch. No font asset is shipped regardless. No theme engine (colors are `#define`s in `render.c`). No animation engine. No image/icon loading (every application that wants an icon has a `TODO` noting this). No text-input widget despite keyboard-routing plumbing existing.

## Window Management

**Current window strategy.** Cascading placement only — new application windows are offset 30px diagonally from the previous one by `luna-shell`, wrapping back to a base position after a fixed bound. There is no user-initiated move or resize (no drag handling exists anywhere in the input-routing code), and `SET_STATE` (minimize/maximize/fullscreen) is accepted by the compositor but explicitly not acted on (`/* In v0.1, we just accept state changes... */` — the comment's own version label is itself stale, predating this milestone).

**Focus model.** Two-tier, as described in §LunaGUI: per-widget focus inside a single application's own tree, and per-surface compositor focus arbitrated by either click-to-focus or the WM's `SET_FOCUS`. Alt+Tab is implemented as array rotation in `luna-shell`, not as a compositor-native concept — the compositor has no notion of a focus stack independent of whatever the WM last told it.

**Surface ownership.** Unchanged from the previous milestone: surfaces belong to the client session that created them; only that session (or, newly, a capability-granted WM session) can mutate them.

**Shell responsibilities.** Placement policy, focus policy, global hotkey handling, wallpaper, and a status/top bar — all implemented in `luna-shell` as userland policy, consistent with the compositor's mechanism-not-policy design.

**Compositor responsibilities.** Mechanism only: surface storage, Z-ordered compositing, capability gating, and now WM-message relay/application. The compositor does not itself decide where windows go or which one is focused absent WM/client instruction.

**Future work.** Drag-to-move and edge/corner resize (no input plumbing exists for either); real `SET_STATE` behavior (minimize should presumably move a surface to a hidden/non-composited state, maximize should resize to display bounds); a visual focus indicator (no border or dimming differentiates the focused window today); reconciling `luna-shell` and `luna-desktop` into one shell implementation; bounds-checked `SET_SURFACE_POSITION`.

## Engineering Decisions

**DL-053 (carried over, not new this milestone)** — the 6-byte TLV header (2-byte type, 4-byte length), accepted last milestone, remains the canonical LGP wire format and is what the new WM and clipboard message families build on.

**No new Decision Log entry exists for the window-manager protocol extension.** The DCKL decision log's highest accepted entry is still DL-053. The six new `WM_*` message types, the new `LGP_CAP_WINDOW_MANAGER` capability bit, the click-to-focus-versus-WM-focus arbitration model, and the choice to encode WM messages little-endian against the rest of the protocol's big-endian convention are all real architectural decisions with no corresponding entry. This is notable because the project's own stated methodology is "documentation-first" (architecture decisions enter the DCKL before code), and DL-053 itself is presented in this same repository as a model example of catching and formally resolving exactly this kind of implementation-vs-decision-log drift. The WM protocol is the first instance this audit found of that discipline lapsing rather than being followed.

## Performance

**Software rendering throughout.** No GPU acceleration anywhere in the stack — the compositor blits surfaces into a CPU-mapped dumb buffer, and LunaGUI renders widget trees into CPU-mapped shared memory. This is unchanged from the prior milestone and was an explicit, documented sequencing decision (software path before Vulkan, per DL-026).

**Full repaint on every commit.** `lgp_surface_manager_composite()` recomposites every visible surface from scratch on every `COMMIT_BUFFER`, with no damage-region tracking. `lgui_widget_render()` likewise re-renders an entire window's widget tree on every `dirty` flag, with no per-widget invalidation. For the current application complexity (small widget counts, infrequent updates) this is unlikely to be observable, but it does not scale with window count or update frequency.

**Memory model.** Fixed-size, stack/static allocation throughout the new code — 32-child widget arrays, 256-byte text buffers, 16-entry WM key-grab tables, 64-surface compositor limits, 40-entry file-listing caps. No widget tree or surface set in this release can grow beyond these compile-time bounds; exceeding them silently truncates rather than erroring (e.g., `luna-files` caps at 40 entries with an overflow-count indicator, but most other fixed arrays have no equivalent signal).

**Single-threaded event loops.** LunaGUI's `poll()` loop runs at a fixed 16ms (~60Hz) timeout regardless of whether any event is pending. The compositor's `epoll_wait()` loop blocks indefinitely (`-1` timeout) and only does work on actual fd readiness, which is the more efficient of the two designs already in the codebase.

**Known bottlenecks.** Full-tree re-render and full-surface recomposite, as above; the compositor's surface lookup, hit-test, and WM-relay paths are all linear scans over fixed-size arrays (`LGP_SURFACE_MAX`, `state->clients`), acceptable at current scale but not designed for growth.

**Optimization opportunities not yet pursued.** Damage-rectangle tracking for both the compositor's composite pass and LunaGUI's render pass; event-driven (rather than fixed-interval) LunaGUI polling; replacing linear surface/client scans with indexed lookup once surface counts grow beyond a handful.

## What Was Learned

**The async, degrade-gracefully service-supervision model designed in v0.1 paid off under real load.** `lgp-compositor` is deliberately excluded from the Stage-4 dependency-graph pass and started manually in Stage 5; `luna-shell` is an ordinary graph-dependent service that only becomes startable once `lgp-compositor`'s readiness socket appears. This milestone added a real consumer of that pattern (`luna-shell` depending on `lgp-compositor`) without needing to change the supervisor itself, which is reasonable evidence the original design generalizes.

**Per-file `-Werror` builds have a blind spot at translation-unit boundaries, and this milestone found it.** The `lgui_font_draw_text` signature mismatch exists specifically because no single file sees both the declaration and the conflicting definition — `-Wall -Wextra -Wstrict-prototypes -Wmissing-prototypes`, all enabled project-wide, catch nothing here, because each file is individually internally consistent. This is a real, generalizable lesson for the project: a private header shared across files needs every file that defines or calls through it to actually include it, and nothing in the current toolchain enforces that automatically (LTO, or a stricter shared-header discipline, would have caught this).

**Feature velocity outpaced build-system maintenance within a single milestone, twice, in the same way.** Both the compositor's `Makefile.inc` and LunaGUI's `Makefile.inc` have the identical failure mode: a new `.c` file was added to the source tree and never added to the corresponding `*_SOURCES` variable. This happened independently in two different subsystems within the same body of work, which suggests it is a process gap (no build-after-every-file-add discipline, and CI not actually exercising `make all` until the very commit that changed Gate 1 to do so) rather than a one-off mistake.

**The documentation-first discipline that produced DL-053 did not repeat for the WM protocol extension.** This is worth stating plainly because the project holds itself to that standard explicitly and visibly (the DCKL, the decision log, the prior milestone's self-audit reports). The WM protocol's design choices — capability bit allocation, little-endian framing, click-vs-WM focus arbitration — are reasonable on inspection, but they are not currently recorded anywhere a future contributor would find them before reading source.

**Lessons carried into v0.3.** Build verification needs to happen before a milestone is declared complete, not assumed from the fact that individual files were written correctly. A shared private header needs an explicit "who must include this" contract. Status documents (`IMPLEMENTATION_STATUS.md` and equivalents) should be regenerated from the actual final commit of a milestone, not from the second-to-last one.

Based on what currently exists in the repository, the realistic next-milestone scope is:

1. **Fix the build.** This is not optional groundwork for some other feature — it is the prerequisite for every other item below being verifiable at all. Concretely: include `widget_private.h` in `core/application.c` and `core/window.c`; add the five orphaned source files to their respective `Makefile.inc` source lists; reconcile `lgui_font_draw_text`'s declaration and definition to a single signature; ship a real `.psf` font asset and install it via `build-image.sh`.
2. **Close the deployment gap.** Add the remaining eight application binaries to `build-image.sh`'s copy list so they exist on a booted system once they're launchable.
3. **Connect keyboard input to a real consumer.** The compositor-to-widget plumbing exists; nothing uses it yet. The highest-leverage target is `luna-terminal` (PTY input) since the ANSI/PTY half of that path is already complete, followed by a basic text-input widget for `luna-settings` and similar forms.
4. **Reconcile `luna-shell` and `luna-desktop`** into one shell implementation, or explicitly retire one with a recorded decision.
5. **Window move and resize.** No input plumbing exists for either today; this is new work, not a fix.
6. **CI coverage for the code added this milestone.** At minimum, get `make all` actually passing in CI again (it does not currently), and add unit coverage for `wm.c`'s encode/decode functions, mirroring the existing `lgp-compositor` protocol test pattern.
7. **A Decision Log entry for the WM protocol**, written after the fact if necessary, documenting the capability/endianness/focus-arbitration choices already made in code.

What should not be started yet, on the same reasoning the project applied last milestone: the installer's disk-write backend (contained, well-scoped, but not blocking anything else — can wait), `luna-ai-d` and `lpkg` (both remain zero-code and have no compositor/shell substrate stable enough to build against, which is now even more true given the build is currently broken), and any atomic-KMS/Vulkan work (still correctly sequenced after the software path, which is itself not yet runtime-verified).
